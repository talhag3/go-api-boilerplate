# Go API Boilerplate

Hey! This is a simple user management REST API I built in Go. It's my first real project using Go (I'm coming from a Node.js/Express background), so I tried to structure it using clean architecture / DDD principles to make sure it's scalable. 

I'm still getting used to Go's package structure, so let me know if anything looks weird!

## What's under the hood?

I wanted to avoid massive ORMs like GORM because I prefer writing raw SQL, so I chose:
* **Fiber v3**: Feels a lot like Express/NestJS. It's super fast and easy to set up.
* **PostgreSQL** with **pgx**: The standard postgres driver for Go.
* **sqlc**: This tool is amazing. I write SQL in `internal/db/queries/users.sql`, and it generates all the Go code for querying. No more writing string queries and scanning rows manually.
* **Goose**: Used for database migrations. It's configured to run automatically when the app starts.
* **slog**: Go's built-in structured logging package.

## Project Structure

I tried to keep things decoupled following clean architecture:
```text
├── cmd/
│   └── api/
│       └── main.go         # Application entry point (composition root)
├── internal/
│   ├── config/             # Config loading from .env via godotenv
│   ├── db/                 # DB connection pool setup & migrations
│   │   ├── migrations/     # Goose migration files
│   │   ├── queries/        # SQL queries for sqlc
│   │   └── sqlc/           # Autogenerated sqlc code (don't touch!)
│   ├── domain/             # Pure business models (User struct)
│   ├── handler/            # HTTP handlers (routing & request/response parsing)
│   ├── logger/             # slog initialization
│   ├── middleware/         # Custom middlewares (Logger and Recover)
│   ├── repository/         # Database persistence logic (talks to sqlc)
│   └── service/            # Business logic and validation rules
└── sqlc.yaml               # sqlc configuration
```

*Note: I realized after coding that I accidentally put the `Recover` middleware code inside `middleware/logger.go` and the `Logger` middleware code inside `middleware/recover.go`. Oops! They still work fine since they are imported properly, but I'll rename the files when I have some free time.*

## Getting Started

### 1. Prerequisites
* Go 1.26+ installed
* PostgreSQL database running locally (or via Docker)

### 2. Configuration
Copy the `.env.example` file to `.env`:
```bash
cp .env.example .env
```
Open `.env` and fill in your database credentials:
```env
APP_PORT=8080
APP_ENV=development
DB_HOST=localhost
DB_PORT=5432
DB_USER=your_postgres_user
DB_PASSWORD=your_postgres_password
DB_NAME=go_api_boilerplate
DB_SSLMODE=disable
DB_MAX_CONNS=10
```

### 3. Run the App
To start the server, just run:
```bash
go run cmd/api/main.go
```
The migrations will run automatically on startup, and the server will start listening on the port configured in `.env` (default is `8080`).

## API Endpoints

All endpoints are prefixed with `/api/v1`.

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/health` | Server health check (returns status: ok) |
| `POST` | `/api/v1/users` | Create a new user |
| `GET` | `/api/v1/users` | List users (paginated, e.g. `?page=1&page_size=20`) |
| `GET` | `/api/v1/users/:id` | Get user by UUID |
| `PUT` | `/api/v1/users/:id` | Update user details (supports partial updates) |
| `DELETE` | `/api/v1/users/:id` | Delete a user |

### Example Request (Create User)
**POST** `http://localhost:8080/api/v1/users`
```json
{
  "full_name": "John Doe",
  "email": "john.doe@example.com"
}
```

### Example Response (Success)
```json
{
  "success": true,
  "data": {
```
